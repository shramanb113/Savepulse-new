Executive Summary SavePulse is a real-time emergency response platform that unifies patients, ambulances, and hospitals in a single coordinated flow. India&apos;s emergency medical ecosystem suffers not from a shortage of resources, but from a catastrophic lack of coordination at the moment it is needed most. SavePulse addresses this by eliminating fragmented phone calls, stale hospital listings, and sequential ambulance dispatch — replacing them with instant parallel dispatch, live tracking, and deferred payment. The platform is designed around a single insight: most healthcare apps assume users are calm. SavePulse is designed for panic. Problem Fragmented emergency response — patients cannot find available hospitals or ambulances quickly enough Solution Real-time coordination app connecting patients, hospitals, and ambulance drivers in one unified flow Core Innovation Parallel ambulance dispatch + hospital-readiness matching + deferred payment with 48-hour grace period Target Market India — urban and semi-urban areas with fragmented emergency infrastructure Hackathon Context 36-hour build — MVP scoped to core dispatch, tracking, and coordination flows 2. Problem Statement 2.1 The Core Problem In an emergency, every second counts. Yet patients and families are forced to: Call multiple hospitals manually to check bed or ICU availability Rely on stale online listings that do not reflect real-time capacity Call ambulance services sequentially, waiting for each to respond before trying the next Navigate payment barriers mid-emergency, causing dangerous delays Coordinate between ambulances, hospitals, and family contacts with no unified view 2.2 Observed from UI Screenshots Analysis of the SavePulse prototype screens reveals the following user pain points being addressed: Landing Page (Emergency Search Screen) Users need to specify emergency location and toggle between Hospital and Ambulance modes — indicating the current system requires manual selection of service type rather than auto-matching Hospital cards display ER wait time and trauma level, confirming that raw proximity alone is insufficient for emergency routing The dispatching ambulance callout on the map showing ETA and turn-by-turn instruction addresses the core anxiety of &apos;is help actually coming?&apos; SOS Professional Dashboard Emergency Category selector (Cardiac Arrest, Accident/Trauma, Respiratory, Maternal) confirms that emergency type must drive matching logic, not just distance The Hospitals / Ambulances toggle indicates that the same dashboard serves dual lookup purposes, which creates cognitive load in a crisis scenario ER Wait time shown alongside distance on hospital cards is a strong UX decision — nearest is not always fastest 3. API Endpoints Method Endpoint Description Key Logic POST /api/match-hospitals Returns top 5 hospitals matching emergency type and proximity SQL filters by emergency type flags + beds_available &gt; 0, orders by distance formula POST /api/emergency Creates emergency request and triggers parallel dispatch Inserts row with status=pending, calls dispatchAmbulances() in background, returns immediately POST /api/accept-request Hospital accepts a request PostgreSQL transaction — UPDATE WHERE status=pending, rowCount check POST /api/create-payment-intent Creates Stripe deferred payment PaymentIntent with capture_method=manual, saves intent ID to DB GET /api/request/:id Fetches current status of an emergency request Used by tracking screen to sync initial state on load PATCH /api/hospital/availability Hospital updates bed/ICU counts Direct update to hospitals table — triggers Socket.io broadcast to relevant users 10.
